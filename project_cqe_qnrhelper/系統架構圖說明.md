# AI 問卷自動填答系統－系統架構圖說明

## 一、系統架構定位

本系統的目標，是讓工程師上傳車廠問卷後，由 AI 自動完成：

- 問題解析
- 知識檢索
- 預填回答
- 證據引用
- 信心度評估
- 人工審查

此系統不是完全取代工程師，而是提供「AI 預填 + 工程師確認」的工作模式。

## 三、系統主要模組說明

### 1. 問卷輸入模組

負責接收客戶提供的問卷檔案，支援格式可包括：

- Excel
- Word
- PDF

此模組的主要任務：

- 讀取檔案內容
- 辨識問卷欄位
- 擷取題目、說明、填答欄位
- 保留原始格式資訊，方便後續回填

## 四、問卷解析模組

此模組負責把非結構化或半結構化問卷，轉成可處理的標準資料格式。

### 輸出欄位建議

- Question ID
- Question text
- Question category
- Customer name
- Source file
- Sheet name / page number

### 模組功能

- Excel 欄位定位
- Word / PDF 文字擷取
- 題目切分
- 多題型辨識（Yes/No、敘述題、表格題）

## 五、問題理解模組

此模組負責理解題目真正要問什麼，並轉成適合檢索的查詢語句。

### 功能重點

- 判斷題目主題
- 抽取關鍵字
- 辨識問句意圖
- 建立檢索 query

### 例子

原始題目：

> Do you have a warranty/NTF handling process?

可轉換為：

- warranty process
- NTF handling flow
- complaint / return handling
- after-sales quality procedure

## 六、知識檢索模組（RAG 核心）

這是整個系統最重要的核心模組之一。

AI 不直接憑空作答，而是先從企業知識庫中找出最相關的內部文件，再根據文件內容生成回答。

### 知識來源可包含

- 品質程序文件
- Control Plan
- PFMEA
- APQP 文件
- NTF / Warranty 流程
- SOP / Flow chart
- 過去車廠問卷
- 8D / Lesson learned

### 技術組成

- 文件切塊（Chunking）
- 向量化（Embedding）
- 向量資料庫（FAISS / Chroma / Milvus）
- Metadata 過濾
- Top-K 相似片段召回

### Metadata 建議欄位

- document_name
- document_type
- department
- revision
- section
- owner
- source_path

## 七、回答生成模組

回答生成模組根據「題目」與「檢索到的證據」，輸出標準化答案。

### 輸出格式建議

1. Direct Answer
2. Evidence
3. Confidence

### 範例

```text
Direct Answer:
Yes. Inventec manages warranty-related issues through a formal NTF handling process.

Evidence:
Procedure: AME-QP-045
Section: NTF Handling Flow
Owner: CQE + MP Team

Confidence:
High
```

### 生成原則

- 不可脫離證據亂答
- 回答必須可追溯
- 回答語氣一致
- 無資料時要誠實標示

## 八、無資料時的處理機制

如果系統找不到足夠證據，不應該直接虛構答案，而應該給工程師一個可編修的建議版本。

### 建議輸出

```text
No direct internal reference found.

Suggested response:
Inventec evaluates customer requirements through APQP review
and implements necessary process updates before SOP.

Action recommended:
Confirm with AQE or Quality system owner.
```

### 設計目的

- 避免 AI 幻覺
- 降低工程師從零撰寫負擔
- 保留人工專業判斷空間

## 九、審查介面模組（Review UI）

此模組提供工程師進行最終審查。

### 工程師需要看到的欄位

- Question
- AI answer
- Evidence
- Confidence

### 工程師操作

- Approve
- Edit
- Reject

### UI 設計原則

- 一題一列
- 回答與證據同畫面顯示
- Low confidence 顯著標示
- 可快速修改後確認

## 十、輸出與回填模組

審查完成後，系統需支援將結果輸出成客戶需要的格式。

### 輸出形式

- 回填 Excel 問卷
- 匯出審查報表
- 產出 evidence list
- 匯出問答紀錄供知識庫累積

## 十一、資料流說明

```text
問卷檔案
→ Parser 解析
→ 題目結構化
→ Query 生成
→ 企業知識庫檢索
→ LLM 生成回答
→ 顯示 Evidence / Confidence
→ 工程師審查
→ 匯出最終答案
```

## 十二、建議技術架構

### 前端

- Streamlit（MVP 最快）
- 或 Web UI（React / Next.js）

### 後端

- Python
- FastAPI

### AI 能力

- OpenAI API
- Embedding model
- LLM answer generation

### 文件處理

- pandas / openpyxl
- python-docx
- PDF parser

### 向量資料庫

- FAISS（MVP）
- Chroma
- Milvus（擴充版）

## 十三、MVP 架構建議

如果先做第一版，建議先聚焦在最小可行流程：

```text
Excel 問卷上傳
→ 擷取問題
→ 搜尋企業文件
→ 生成回答
→ 顯示審查畫面
→ 匯出 Excel
```

### MVP 優先功能

1. 只支援 Excel 問卷
2. 只支援單一知識庫資料夾
3. 只產出 AI answer + evidence + confidence
4. 只提供人工確認後匯出

## 十四、後續擴充方向

### Phase 1

- AI 自動填寫問卷

### Phase 2

- AI 產出 Audit evidence package
- 自動整理對應程序、表單、紀錄

### Phase 3

- 建立 OEM questionnaire library
- 累積標準問答模板
- 提高未來新問卷自動回答率

## 十五、架構設計原則

這套系統在車用品質場景中，最重要的不是「回答快」，而是「回答可被稽核接受」。

因此設計原則應為：

- 先找證據，再生成答案
- 每一題都必須可追溯來源
- 無資料時不可亂編
- 工程師保有最終決定權
- 回答結果可以持續沉澱為企業知識資產

## 十六、總結

這套架構本質上是一個結合：

- 文件解析
- 企業知識庫
- RAG 檢索
- LLM 回答生成
- 人工審查流程

的 AI 問卷自動填答系統。

最適合先從車廠 Self-Assessment / Supplier Audit Questionnaire 場景切入，先做出 MVP，再逐步擴充到 evidence package 與 OEM 問卷知識庫。
